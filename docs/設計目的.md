# JSX Viewer 設計目的

## 問題描述

使用 Claude Code 時，經常會請 AI 產生**視覺化的 JSX 說明文件**。這些檔案不是普通的文字文件，而是完整的 React 元件，包含：

- 動畫效果（CSS keyframes、IntersectionObserver 觸發動畫）
- 互動元素（useState 控制的 Tab 切換、hover 效果）
- 資料視覺化（對比條、統計數字動畫）
- 精心設計的深色主題排版

**問題在於：** 這些 `.jsx` 檔案無法直接開啟。你需要：

1. 建立一個 React 專案
2. 安裝��賴
3. 把檔案放進去
4. 修改路由或入口點
5. 啟動開發伺服器

每次想看一個新的 JSX 說明檔案，都要重複這個過程。這嚴重降低了「AI 產生視覺化文件」的實用性。

## 解決方案

JSX Viewer 的核心理念是 **「拖放即渲染」** —— 把上面 5 個步驟簡化為 1 個動作。

## 設計決策與取捨

### 決策 1：為什麼用 Babel Standalone 而不是 SWC？

| 考量 | Babel Standalone | SWC (WASM) |
|------|-----------------|------------|
| 穩定性 | 極高，久經考驗 | 較新，偶有邊界問題 |
| 體積 | ~800KB | ~2MB |
| 速度 | 夠用（單檔編譯 <100ms） | 更快但 MVP 不需要 |

**結論：** MVP 階段穩定最重要。800KB 對本地開發工具完全可以接受。未來如果效能成為瓶頸，可以切換到 SWC。

### 決策 2：為什��用 iframe 而不是直接渲染？

Claude Code 產生的 JSX 檔案經常包含全域 CSS：

```jsx
<style>{`
  * { box-sizing: border-box; margin: 0; padding: 0; }
  @keyframes pulse { ... }
  ::-webkit-scrollbar { width: 6px; }
`}</style>
```

如果直接渲染在 Viewer 的 DOM 中，這些全域樣式會**破壞 Viewer 自身的介面**。

iframe 提供完全隔離：
- CSS 隔離（全域樣式不會洩漏）
- JavaScript 隔離（全域變數不會衝突）
- DOM 隔離（元素操作不會影響 parent）

### 決策 3：為什麼用 Import Maps 而不是 URL 改寫？

處理 `import { motion } from "framer-motion"` 有兩種方式：

**方式 A：URL 改寫**
```javascript
// 把 source code 中的 import 改成完整 URL
import { motion } from "https://esm.sh/framer-motion";
```

**方式 B：Import Maps（我們選擇的方式）**
```html
<script type="importmap">
  { "imports": { "framer-motion": "https://esm.sh/framer-motion" } }
</script>
```

Import Maps 更好因為：
- 原始碼不需要被改寫（更乾淨）
- 自動處理子路徑 import（`package/sub`）
- 自動處理 dynamic `import()` 呼叫
- 是瀏覽器原生標準

### 決策 4：為什麼 Babel 在 parent 編譯而非 iframe 中？

- 只載入一次 Babel（800KB），不用每個 iframe 都載入
- 直接在 parent 捕獲語法錯誤，不需要額外的 postMessage 通訊
- 編譯後的純 JS 注入 iframe 更快

### 決策 5：為什麼用 `?external=react,react-dom`？

當從 esm.sh 載入第三方套件時，如果不加 `external` 參數，每個套件會各自帶一份 React。這會導致：

```
Error: Invalid hook call. Hooks can only be called inside of the body
of a function component. This might happen because of:
1. Mismatching versions of React and the renderer
```

加上 `?external=react,react-dom` 後，esm.sh 不會打包 React，而是留下 bare import，由 Import Map 統一解析到同一個 React 實例。

## 設計原則

### 最小可行性（MVP）

每個功能都問：「沒有這個，使用者能完成核心任務嗎？」

- 能拖放檔案並看到渲染結果 → **必要**
- 能同時看多個檔案 → **值得做**（成本低，體驗提升大）
- 能編輯檔案後即時更新 → **未來再做**（需要 CodeMirror 等編輯器）
- 能瀏覽��錄結構 → **未來再做**（需要後端）

### 為擴充而設計

雖然是 MVP，但架構預留了擴充空間：

```
現在                           未來可擴充
────                           ────────
拖放載入           →           + 目錄瀏覽（加後端）
唯讀檢視           →           + 內建編輯器（CodeMirror）
單一 React 版本     →           + 版本切換
僅 JSX             →           + TypeScript 支援
只支援 React        →           + Vue/Svelte 元件
```

### 零配置

使用者不需要了解 Babel、Import Maps 或 esm.sh —— 這些都在背景自動處理。拖放一個檔案，看到結果，就這樣。
